# 09 вҖ” Audio Mastering Pipeline

The end-to-end flow from "user drops a file" to "user gets a mastered file."

> **Pipeline upgraded in 2026.** What used to be "Matchering with `limiter_mode`" is now a real chain: optional AI noise reduction вҶ’ Matchering вҶ’ polish (HPF + de-ess + leveling) вҶ’ LUFS normalize вҶ’ true-peak limiter. User-facing setting is now `loudness_target` (LUFS-based), not the old `limiter_mode`. See [#users-loudness-knob](#user-facing-loudness-knob) below.

## Players

| Component | Role |
|---|---|
| Browser (Next.js client) | UI, file selection, upload, polling, download |
| Modal FastAPI ([backend/modal_app.py](../backend/modal_app.py)) | Receives the upload, runs the chain, posts back to Next.js |
| Cloudflare R2 | Object storage for raw uploads + mastered outputs (free tier) |
| Vercel Blob | Permanent storage for subscriber outputs |
| Next.js webhook ([`/api/webhooks/job-complete`](../src/app/api/webhooks/job-complete/route.tsx)) | DB writes + email send |
| Postgres | Tracks notifications, premium jobs, usage, free-tier file metadata |

## The job ID

Every master gets a `jobId` (a UUID generated by Modal in `POST /master`). Same handle across:
- Client polls `GET /status/{jobId}`.
- Webhook arrives keyed by `jobId`.
- `JobNotification.jobId`, `PremiumUserJob.jobId`, `FreeUserFile.jobId` all link to it.

## Loudness targets

| `loudness_target` | Integrated LUFS | When to use |
|---|---|---|
| `conservative` | **-16 LUFS** | Apple Podcasts spec. Dialog-heavy shows, audiobook style. Keeps more dynamics. |
| `standard` | **-14 LUFS** | **Default.** Spotify's normalization target. Sounds full and consistent on Spotify and most streamers. |
| `loud` | **-12 LUFS** | YouTube / broadcast-leaning. Punchy. Can sound slightly compressed but cuts through noisy environments. |

True-peak ceiling on all targets is **-1 dBTP** (pedalboard's `Limiter` at -1 dB).

Real-world final LUFS lands ~0.5 dB below target because of the limiter doing its job вҖ” we overshoot the makeup gain by +0.5 dB to compensate. So `standard` outputs at roughly -14.3 LUFS in practice.

## User-facing loudness knob

The API parameter is **`loudness_target`** (string: `conservative` | `standard` | `loud`).

### The legacy `limiter_mode` param
- Still accepted for one deploy cycle.
- If sent without `loudness_target`, it maps via:
  ```
  gentle вҶ’ conservative
  normal вҶ’ standard
  loud   вҶ’ loud
  ```
- After all clients have shipped the new param, remove the mapping. The `LEGACY_LIMITER_MODE_MAP` dict in [backend/modal_app.py](../backend/modal_app.py) is the single source of truth.

### Why the old `limiter_mode` was broken
The old code mapped the user's choice directly to Matchering's `threshold` parameter:
```python
"gentle": 0.95, "normal": 0.9, "loud": 0.8
```
Matchering's `threshold` is the **peak limiter ceiling**, not a loudness target. Setting it to `0.8` caps peaks at -1.94 dBFS вҖ” *quieter* than `0.95`. RMS was set by the reference's RMS and was largely independent of `threshold`. So "Loud" was actually quieter than "Gentle" вҖ” backwards from user expectations.

The new design moves loudness control out of Matchering's threshold and into an explicit LUFS measurement + makeup gain stage after Matchering.

## AI noise reduction (optional)

Toggle on via `noise_reduction=true`. Uses [`noisereduce`](https://github.com/timsainb/noisereduce) вҖ” spectral noise reduction via STFT.

Settings:
- `stationary=False` (adaptive вҖ” works for room tone that changes)
- `prop_decrease=0.75` (aggressive enough to be audible, gentle enough to not dull the voice)
- `n_fft=1024`, `time_constant_s=2.0`

Runs BEFORE Matchering so the spectral analysis sees the cleaned signal. Adds ~30-90 seconds for a typical 30-minute podcast.

Default: **off**. Turning this on can subtly affect voice quality (very mild "underwater" effect on very noisy sources), so we don't surprise users with it.

## Stage-by-stage walkthrough

### Stage 0 вҖ” Pre-flight checks (browser)

Before kicking off a master, the client ([HomeClient](../src/components/HomeClient.tsx) / [MasteringTool](../src/components/MasteringTool.tsx)) gates on:

```ts
GET /api/rate-limit/check?userId=...    // 2/week limit
GET /api/subscription/status            // for premium UX
GET /api/hq-purchase/status             // credits state
GET /api/storage/check                  // 5GB cap check for subs
```

If a 24-bit master is requested by a non-subscriber, the client also calls `POST /api/hq-purchase/status` (decrement credit).

### Stage 1 вҖ” Upload to Modal

Browser uploads file `multipart/form-data` to Modal's `/master` endpoint. Query params:
```
target_file_id=<uuid>
template_id=voice-optimized | male-podcast | female-podcast | news-broadcast
output_quality=standard | high      (16-bit | 24-bit)
loudness_target=conservative | standard | loud
noise_reduction=true | false
```

Modal:
1. Saves the upload to a temp file inside its container.
2. Uploads it to R2 at `uploads/{jobId}/{filename}`.
3. Generates a `jobId`.
4. Returns `{ jobId }`.
5. Kicks off the actual mastering as a Modal background function.

### Stage 2 вҖ” Noise reduction (optional)

If `noise_reduction=true`, load the audio with `soundfile`, transpose to `(channels, samples)`, call `noisereduce.reduce_noise(...)`, save back to a clean WAV. Matchering then reads the clean WAV.

### Stage 3 вҖ” Matchering

```python
podcast_config = mg.Config(
    max_length=635_040_000,   # ~4 hours @ 44.1 kHz
    threshold=0.95,           # leave headroom for the post-chain
)
mg.process(
    target=matchering_input,
    reference=reference_path,
    config=podcast_config,
    results=[mg.pcm24(matched_path)],
)
```

Intermediate is 24-bit so we don't bottleneck precision before the loudness stage. Reference files live in the container at `/references/*`. The `_pick_reference_path()` helper prefers `.wav` if present, falls back to `.mp3`. See the [reference re-mastering script](../backend/scripts/remaster_references.py) for upgrading `.mp3` вҶ’ `.wav` references.

### Stage 4 вҖ” Polish chain

```python
polish_chain = Pedalboard([
    HighpassFilter(cutoff_frequency_hz=40.0),
    PeakFilter(cutoff_frequency_hz=200.0,  gain_db=-1.0, q=0.7),
    PeakFilter(cutoff_frequency_hz=2800.0, gain_db= 1.0, q=0.8),
    PeakFilter(cutoff_frequency_hz=6500.0, gain_db=-2.5, q=2.5),
    Compressor(threshold_db=-18.0, ratio=2.0, attack_ms=8.0, release_ms=100.0),
])
```

Runs AFTER Matchering so it doesn't fight Matchering's spectral matching. Pedalboard expects `(channels, samples)`; the code transposes both ways.

### Stage 5 вҖ” LUFS normalize + true-peak limit

```python
meter = pyln.Meter(sr)
current_lufs = meter.integrated_loudness(audio)
target_lufs = LOUDNESS_TARGETS[loudness_target]
gain_db = (target_lufs - current_lufs) + 0.5   # +0.5 dB overshoot

finalize_chain = Pedalboard([
    Gain(gain_db=gain_db),
    Limiter(threshold_db=-1.0, release_ms=100.0),
])
```

`pyloudnorm` implements ITU-R BS.1770-4. Integrated loudness вҖ” averages over the whole file, weighted by gating thresholds вҖ” is the metric platforms use.

### Stage 6 вҖ” Write output

```python
subtype = "PCM_24" if output_quality == "high" else "PCM_16"
sf.write(output_path, audio, sr, subtype=subtype)
```

Sample rate is preserved from the source. We don't resample.

### Stage 7 вҖ” Output storage

When the chain finishes, Modal branches:

**Free user (no premium job link):**
- Upload output to R2 at `outputs/{jobId}_mastered.wav`.
- R2 returns a public download URL.
- Update `job_statuses[jobId].download_url`.
- POST webhook to `/api/webhooks/job-complete` with the URL.

**Premium user:**
- Look up `/api/files/get-blob-upload-url` (Bearer auth) to see if we should save to Vercel Blob.
- If yes (quota allows), upload the output bytes directly from the container to Vercel Blob using the returned token.
- Webhook to `/api/webhooks/job-complete` includes `blobData: { url, pathname, size, fileName }`.
- The webhook handler creates a `SubscriberFile` row.

### Stage 8 вҖ” Webhook & email

[`/api/webhooks/job-complete`](../src/app/api/webhooks/job-complete/route.tsx) does:
1. Verify `Authorization: Bearer ${WEBHOOK_SECRET}`.
2. If `blobData` present вҶ’ save `SubscriberFile`.
3. Update `FreeUserFile` (if exists) вҶ’ `status: "completed"`, set `downloadUrl`.
4. Find `JobNotification` by `jobId`. If found and `emailSentAt === null`:
   - Render [MasteringComplete](../src/emails/MasteringComplete.tsx) with `{ downloadUrl }`.
   - Send via Resend.
   - Update `emailSentAt` and `status: "sent"`.

### Stage 9 вҖ” Cleanup

Free-tier files (in R2 only) are deleted by a Modal cron at 00:00 UTC daily вҖ” `cleanup_old_files()` in [backend/modal_app.py](../backend/modal_app.py). Subscriber files persist until the user deletes them.

## Templates explained

| Template | Reference design | When to recommend |
|---|---|---|
| `voice-optimized` | Generic neutral voice mastering | Default. Works for any voice. |
| `male-podcast` | Heavier low-mid presence, fuller production | Male hosts wanting a "broadcast" sound |
| `female-podcast` | Brighter top end, controlled sibilance | Female hosts |
| `news-broadcast` | Tightly limited, news-radio vibe | News-style shows |

Reference files live at `/references/*.{wav,mp3}` inside the Modal container (baked at image build time via `add_local_dir("references", "/references")`). The helper `_pick_reference_path()` automatically picks the `.wav` version if it exists.

To replace a reference:
1. Drop a new audio file at `backend/references/<template-id>.wav` (or `.mp3`).
2. Run [`scripts/remaster_references.py`](../backend/scripts/remaster_references.py) to normalize loudness.
3. `modal deploy modal_app.py`.

## Output quality

- **16-bit** вҖ” default for everyone. PCM 16-bit WAV, source sample rate preserved.
- **24-bit** вҖ” gated. Subscribers always; non-subs need an HQ credit.

24-bit gives more dynamic-range headroom and is what platforms like Spotify prefer for upload.

## Limiter behavior

True-peak ceiling is always **-1 dBTP** (pedalboard's `Limiter`, sample-peak limiter вҖ” not strictly true-peak but close enough for podcast use cases). Release time: 100ms.

The actual loudness reduction from the limiter depends on the polish-chain output level relative to the target. Most podcasts hit the target with 0.5вҖ“2 dB of limiter action.

## Failure modes

| Symptom | Cause | What the code does |
|---|---|---|
| Modal container OOMs on a giant file | Audio too large for 8 GB RAM | Container memory bumped to 8 GB; long files (>~4hr) error at Matchering's `max_length` check |
| Network drop during upload | User's connection | Browser shows error, no `UsageLog` row was written yet вҖ” retry without burning rate-limit credit |
| Webhook arrives but `JobNotification` missing | User never subscribed | Webhook silently skips email вҖ” that's fine |
| Webhook arrives twice | Modal retry | `emailSentAt` check makes the email path idempotent; `SubscriberFile` create is idempotent because the upload pathname is unique |
| `pyloudnorm` returns `-inf` (silent input) | Audio is digital silence | Fallback to -40 LUFS so the gain calculation doesn't blow up |
| Noise reduction creates "underwater" artifact | `prop_decrease` too aggressive for the source | Lower `prop_decrease` (currently 0.75) вҖ” but this is a rare complaint |
| Final LUFS is 1+ dB below target | Polish chain reduced the signal more than expected | The +0.5 dB overshoot helps but isn't infinite. Could add a 2-pass measure-and-correct loop if it becomes an issue |
| Matchering's reference is a quiet MP3 | Reference loudness propagates to output | No longer matters вҖ” LUFS stage normalizes regardless of reference loudness |

## Performance

For a typical 30-minute podcast at 44.1 kHz stereo on Modal (4 CPU, 8 GB RAM):

| Stage | Time |
|---|---|
| Download from R2 | 1вҖ“3 s |
| Noise reduction (if enabled) | 30вҖ“90 s |
| Matchering | 30вҖ“90 s |
| Polish + LUFS measure + limit | 10вҖ“20 s |
| Write WAV | 2вҖ“5 s |
| Upload to R2 + Blob | 5вҖ“15 s |
| **Total** | **80вҖ“230 s** (1.5вҖ“4 min) |

Without noise reduction: **50вҖ“140 s**.

## What is NOT in the audio path

- No `ffmpeg` re-encoding at the Next.js layer.
- No streaming. Upload is one-shot multipart, download is one-shot GET.
- No partial / incremental output.
- No client-side mastering.
- No multi-band compression (we have a single broadband leveling compressor + frequency-targeted EQ filters вҖ” enough for voice without the complexity).
- No saturation/harmonic enhancement stage. Could be added later if users want a "warmer" character.
